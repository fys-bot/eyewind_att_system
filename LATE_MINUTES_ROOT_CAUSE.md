# 迟到分钟数问题的根本原因

## 核心问题

经过深入调试，发现问题的根本原因是：**钉钉API返回的打卡记录中，`workDate` 字段不是工作日期，而是打卡时间**。

## 证据

从日志可以看到：
```
日期: '2026-02-01', 打卡时间: '2/2/2026, 9:13:26 AM'
```

这说明：
- 员工在 **2月2日上午9:13** 打卡
- 但这次打卡是为了 **2月1日** 的考勤（跨天打卡）
- 钉钉返回的 `workDate` 字段是 **2月2日**（打卡时间），而不是 **2月1日**（工作日期）

## 为什么计算结果为0

规则引擎计算迟到分钟数的逻辑是：
```typescript
const lateMinutes = Math.max(0, Math.floor((checkInTime.getTime() - workStartTime.getTime()) / 60000));
```

如果：
- `workDate` = 2月1日（错误的工作日期）
- `checkInTime` = 2月2日 9:13（打卡时间）
- `workStartTime` = 2月1日 09:00（根据workDate计算的工作开始时间）

那么：
```
lateMinutes = (2月2日 9:13 - 2月1日 09:00) / 60000
           = 24小时13分钟
           = 1453分钟
```

这是正确的！但如果 `workDate` 被错误地设置为 2月2日：
```
lateMinutes = (2月2日 9:13 - 2月2日 09:00) / 60000
           = 13分钟
```

等等，这也应该是正确的！

让我重新分析...实际上问题可能是：**`attendanceMap` 的 key 是错的**！

## 真正的问题

`attendanceMap` 的结构是：
```typescript
{
  [userId]: {
    [day]: DailyAttendanceStatus
  }
}
```

如果 `day` 是从打卡记录的 `workDate` 字段解析出来的（2月2日），那么：
- 2月1日的打卡记录被放在了 `attendanceMap[userId][2]` 中
- 当我们遍历 `attendanceMap[userId][2]` 时，我们认为这是2月2日的考勤
- 但实际上这是2月1日的考勤（只是在2月2日打的卡）

## 解决方案

我们需要找到 `attendanceMap` 是在哪里构建的，并修复它的逻辑，确保使用正确的工作日期作为 key，而不是打卡时间。

可能的位置：
1. `AttendanceDashboardPage.tsx` - 数据加载和处理
2. `attendanceStatusService.ts` - 后端服务
3. 钉钉API数据处理逻辑

## 临时解决方案

在找到并修复 `attendanceMap` 构建逻辑之前，我们可以：

1. **使用打卡记录的实际时间来判断工作日期**
   - 如果打卡时间在凌晨0:00-6:00，认为是前一天的下班打卡
   - 如果打卡时间在6:00-24:00，认为是当天的上班打卡

2. **或者，直接使用钉钉标记的 `timeResult === 'Late'` 来计算迟到**
   - 如果钉钉标记为迟到，就计算打卡时间与09:00的差值
   - 不再依赖复杂的跨天/跨周/跨月规则

## 建议

最简单的解决方案是：**暂时禁用跨天规则，使用标准的09:00作为迟到基准**。

这样至少能让迟到分钟数显示出来，虽然不够精确，但总比显示0要好。
